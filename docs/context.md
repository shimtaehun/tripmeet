# TripMeet (트립밋) 맥락 노트

## 1. 기술 스택 선정 이유

### 프론트엔드: React Native (Expo)

- **선정 이유:** 1인 개발 환경에서 iOS, Android, 웹을 단일 코드베이스로 지원. Expo 관리형 워크플로우로 네이티브 빌드 환경 구성 없이 개발 가능
- **대안 대비 장점:** Flutter 대비 JavaScript 생태계 활용 가능. 웹뷰 방식 대비 네이티브 성능 확보
- **주의 사항:** 새 네이티브 모듈 도입 시 EAS Build 필요 여부 사전 검토 필요

### 백엔드: Python (FastAPI)

- **선정 이유:** OpenAI SDK와의 궁합이 좋고, 비동기 처리(async/await) 기본 지원. 자동 OpenAPI 문서 생성으로 1인 개발 생산성 확보
- **대안 대비 장점:** Django 대비 경량. Node.js 대비 AI/데이터 라이브러리 생태계 우세

### 데이터베이스: PostgreSQL (Supabase)

- **선정 이유:** 관계형 DB의 정합성 보장. Supabase가 Auth, Storage, 실시간 기능을 번들로 제공하여 인프라 관리 비용 절감
- **Supabase Auth 선정 이유:** 구글 OAuth, JWT 세션 관리를 별도 서버 없이 처리 가능. 무료 티어에서 충분한 MAU 지원

### AI: OpenAI GPT-4o-mini

- **선정 이유:** GPT-4o 대비 약 15배 저렴 (~$0.002/건). 여행 일정 생성 품질은 충분한 수준
- **비용 방어:** Upstash Redis 캐싱을 통해 동일 조건 재호출 시 API 비용 0원

### 실시간 채팅: Firebase Firestore

- **선정 이유:** MVP 단계에서 WebSocket 서버 직접 운영 없이 실시간 채팅 구현 가능. 무료 티어 내 처리 가능
- **이후 전환 계획:** 트래픽 증가 시 FastAPI WebSocket 서버로 마이그레이션 검토

### 이미지 저장: Cloudflare R2

- **선정 이유:** 이그레스(데이터 다운로드) 비용 무료. AWS S3와 API 호환. 무료 티어 10GB 저장 + 월 100만 건 요청

## 2. 배포 인프라 구성 이유 (비용 근거)

| 서비스 | 용도 | 비용 | 선정 근거 |
|--------|------|------|-----------|
| Render.com | FastAPI 백엔드 서버 | 무료 → $7/월 | 무료 티어로 MVP 검증 후 유료 전환. 콜드 스타트 허용 가능한 초기 단계 |
| Supabase | PostgreSQL + Auth + Storage | 무료 → $25/월 | 500MB DB + 1GB Storage + 50,000 MAU 무료. MVP 초기에 비용 0원 |
| Cloudflare R2 | 이미지 저장 | 무료 | 이그레스 무료가 핵심. S3는 이그레스 $0.09/GB 발생 |
| Upstash Redis | AI 결과 캐싱 | 무료 | 하루 10,000 요청 무료. AI 캐시 히트율 높이면 GPT API 비용 대폭 절감 |
| Firebase Firestore | 실시간 채팅 | 무료 | 하루 50,000 읽기 / 20,000 쓰기 무료. MVP 채팅 트래픽 충분히 수용 |
| GPT-4o-mini | AI 일정 생성 | ~$0.002/건 | 캐싱으로 실제 호출 건수를 최소화하여 월 수천 원 수준 유지 목표 |

**MVP 예상 월 비용:** 초기 0원 (무료 티어 내) → 사용자 증가 시 최대 약 $32/월

## 3. 특수 개발 원칙: 비용 방어 전략

> 이 원칙은 모든 개발 작업에서 반드시 준수한다.

### 원칙 1: 토큰 방어 - 컨텍스트 강제 압축

코딩 작업을 지시받을 때, 전체 계획서를 다시 읽지 않는다.
`checklist.md`의 특정 항목 1개와 수정이 필요한 파일 1~2개만 제한적으로 읽고 작업을 수행한다.

- 작업 시작 전: checklist.md에서 해당 항목만 확인
- 작업 대상 파일: 최대 2개로 제한
- 전체 plan.md, context.md 재독은 금지

### 원칙 2: 토큰 방어 - QA 에이전트 호출 최소화

파일 하나를 고칠 때마다 스스로 품질 검사나 전체 리뷰를 실행하지 않는다.
단위 작업 중에는 코드 작성에만 집중한다.

- 자동 전체 리뷰 실행 금지
- 작업 단위: 파일 1개 완료 후 보고 후 정지
- 연속 자동 작업 금지

### 원칙 3: 서버비 방어 - LLM API 캐싱

AI 여행 일정 자동 생성 결과는 반드시 Upstash Redis에 캐싱한다.
동일한 조건의 요청이 들어오면 LLM API를 호출하지 않고 캐싱된 데이터를 반환한다.

**캐시 키 설계:**
```
{destination}:{duration_days}일:{travelers_count}명:{budget_range}
예시) 제주도:3일:1명:30만원대
```

**예산 구간화 규칙:**
- 0 ~ 100,000원 미만 → "10만원대 이하"
- 100,000 ~ 199,999원 → "10만원대"
- 200,000 ~ 299,999원 → "20만원대"
- 300,000 ~ 499,999원 → "30만원대"
- 500,000 ~ 999,999원 → "50만원대"
- 1,000,000원 이상 → "100만원 이상"

**구간화 목적:** 25만원과 28만원을 같은 캐시 키로 처리하여 캐시 히트율 극대화

**캐시 TTL:** 7일 (여행 일정 정보는 단기간 내 유효)

**처리 흐름:**
```
요청 수신
  → 캐시 키 생성
  → Redis 조회
    → 히트: Redis 데이터 즉시 반환
    → 미스: GPT-4o-mini 호출 → 결과 Redis 저장 → 반환
```

### 원칙 4: 서버비 방어 - 이미지 클라이언트 압축

커뮤니티나 맛집 리뷰에 사진을 업로드할 때, 반드시 프론트엔드 단에서 이미지를 500KB 이하로 압축한 후 서버로 전송한다.

- 압축 라이브러리: expo-image-manipulator (Expo 기본 제공)
- 압축 기준: 500KB 이하, 최대 가로/세로 1280px
- 적용 위치: 모든 이미지 업로드 진입점 (맛집 리뷰, 커뮤니티 게시글, 프로필 사진)
- 서버 측 추가 처리 없음 (클라이언트 압축만으로 충분)

**목적:** Cloudflare R2 저장 용량 절약 + 업로드 속도 개선 + 서버 대역폭 절감

### 원칙 5: 법적 리스크 회피 - 위치정보 수동 선택

기기 GPS를 자동 수집하지 않는다.
사용자의 명시적 지역 선택 데이터만 활용하여 위치정보법 신고 의무를 우회한다.

- GPS 권한 요청 금지 (MVP 기간 내)
- 위치 등록: 사용자가 검색 또는 목록에서 직접 선택
- 저장 데이터: 지역명 텍스트만 저장 (좌표 저장 금지)
- 근거: 위치정보의 보호 및 이용 등에 관한 법률 제5조 - 개인위치정보 수집 시 방송통신위원회 신고 의무 발생. 수동 입력은 해당 법 적용 대상 외
